## Context

I'm extracting WavePoint Studio — our Human+AI agentic workflow system — into a standalone, white-label framework called `@sherpa/studio`. This framework will power multiple independent projects:

- **`@sherpa/studio`** — the framework itself (the "Rails")
- **Sherpa** (`../sherpa`) — my AI & Digital Transformation consulting company, the first consumer
- **WavePoint** — my astrology platform, the first customer

The framework encompasses: governance (initiatives, proposals, workstreams), execution (task board, planner/worker/judge pipeline, MCP server, worker scripts), feedback (morning review, judge verdicts), agent infrastructure (role catalog, dispatch), and the Studio UI that ties it all together.

A boundary analysis has already been completed. Key finding: **~80% of Studio is already domain-agnostic.** `src/lib/studio/` has zero astrology imports. The MCP server is 100% generic. Only 3 of 80+ components touch domain-specific code. Full analysis in `docs/initiatives/sherpa-framework-extraction/proposal.md`.

### Vector 2: Convention Portability

**Question:** How do you package and distribute *conventions* — not just code?

The framework isn't just npm packages. It's also:
- `.claude/rules/` files (auto-loaded by glob pattern into Claude Code context)
- `.claude/skills/` (workflow protocols like /rr, /integration-review, /plan-tasks)
- `docs/tasks/README.md` (task file YAML schema convention)
- `docs/agents/roles/*.md` (agent role definitions with frontmatter)
- `docs/initiatives/` directory structure convention
- `CLAUDE.md` patterns (the recursive docs structure)

Explore:
- How other AI-native tooling (Cursor rules, Windsurf rules, Claude Code rules) handle shareable conventions
- Whether conventions should be generated by CLI, synced via git subtree, or vendored as templates
- The shadcn/ui model (copy into your project, then own it) vs. the framework model (import and extend)
- How Turborepo generators and Plop.js handle convention scaffolding
- Version management: how do consumers receive convention updates without losing their customizations?

Read for context:
- `.claude/rules/initiative-convention.md` (the governance convention)
- `.claude/rules/worktree-conventions.md` (the isolation convention)
- `.claude/rules/effort-estimation.md` (session-based estimation)

### Vector 3: White-Label Strategy

**Question:** What needs to be parameterized for the framework to serve any project — not just WavePoint or Sherpa?

Explore:
- Which Studio UI components carry branding assumptions (colors, typography, naming)?

- How Notion, Linear, and Height.app handle workspace-level customization

Read for context:

### Vector 4: Cross-Project Synchronization

**Question:** How do multiple independent projects consume framework updates without merge conflicts or drift?

The model: Sherpa is the upstream repo. WavePoint and future projects consume `@sherpa/studio`. When the framework improves, consumers should be able to pull updates.

Explore:
- npm package versioning (semver for code packages)
- git subtree vs. git submodule for convention files
- The Backstage.io model (framework + app template + plugins)
- Nx and Turborepo remote generators for workspace templates
- How Expo handles SDK version upgrades across consumer projects
- The "vendored scaffold" model — copy once, then diverge (shadcn) vs. "living dependency" model — always import latest (npm package)
- Hybrid approach: code as packages (living dependency), conventions as scaffold (copy + customize)
