# Vector 4: Convention Sync & Drift Detection

**Question:** How do systems detect when generated artifacts drift from source files? What CI/build-time patterns exist for keeping assembled context files in sync?
**Agent dispatched:** 2026-03-18

## Findings

**Pattern 1: Regenerate-and-Diff (dominant pattern)**

Run the generator, check if output changed. Battle-tested across ecosystems.

- **Kubernetes** `hack/verify-codegen.sh`: creates temp worktree from HEAD, runs generator, checks `git status --porcelain`. Gold standard. ([kubernetes/hack/verify-codegen.sh](https://github.com/kubernetes/kubernetes/blob/master/hack/verify-codegen.sh))
- **Go ecosystem**: `go generate ./... && git diff --exit-code`. Simplest variant, widely used.
- **GraphQL Code Generator**: built-in `--check` flag generates in memory, compares to disk. Caveat: formatters break this.
- **diff-check GitHub Action**: runs arbitrary command, then `git diff-index --quiet HEAD`.

**Pattern 2: Content-Hash Comparison**

Store hash of inputs, compare without regenerating.

- **Bazel**: cryptographic hash of all inputs — content-addressable, not timestamp-based.
- **Make with SHA256**: composite hash of source files replaces mtime-based staleness.
- **Nix flakes**: entire dependency graph deterministic via cryptographic hashes in `flake.lock`.

**Pattern 3: Embedded Provenance**

- **Go convention**: `// Code generated by <tool>; DO NOT EDIT.` header.
- **.gitattributes**: `linguist-generated=true` collapses files in GitHub PRs.

**Pattern 4: Context Drift Detection for AI Agents**

- **Codified Context paper** (Vasilopoulos, 2026, arxiv 2602.20478): Python session-start hook parses recent git commits against subsystem-to-file mappings, injects warnings when source files change without specification updates. Key finding: agents "trust documentation absolutely" — stale context files are more dangerous than stale human docs.

**Recommended approach for Sherpa: Regenerate-and-diff + embedded hash + pre-commit hook**

1. `--check` mode: regenerate in temp location, diff against committed file
2. Embedded source hash in generated file header (SHA256 of all input files)
3. Pre-commit hook: if `.claude/rules/*.md` staged, run check mode
4. Session-start warning if embedded hash doesn't match live hash

**Do NOT rely on timestamps** — break on git checkout, cache restores, machine transfers. Content hashing is strictly superior.

## Implications

- The generator script should support a `--check` flag for CI/pre-commit validation
- Embed composite SHA256 of source rules in the generated file header — enables lightweight staleness checks without running the generator
- Pre-commit hook is the right integration point for this repo
- Session-start staleness warning is valuable given that agents trust context absolutely
- Content hashing > timestamps for cross-machine reliability
